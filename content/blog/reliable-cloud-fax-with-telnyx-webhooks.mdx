Fax sounds like a relic until you work with healthcare or legal workflows, where it's
still the system of record. Building a modern fax product means hiding all of that
legacy behind a clean API — and webhooks are where the real engineering lives.

## The shape of the problem

A fax has a lifecycle: queued → sending → delivered (or failed). Telnyx reports each
transition via webhooks, and your job is to turn that noisy, at-least-once stream into a
clean, observable state machine.

## Verify every webhook

Treat inbound webhooks as untrusted until proven otherwise. Telnyx signs payloads, so
the first thing the handler does is verify the signature — before parsing anything.

```ts
if (!verifySignature(req.rawBody, req.headers["telnyx-signature-ed25519"])) {
  return res.status(401).end();
}
```

## Make handlers idempotent

Webhooks are delivered **at least once**, which means duplicates are guaranteed. Every
event carries an id; I record processed ids and short-circuit repeats so a redelivery
never double-counts a delivery or double-charges a customer.

- Persist the event id inside the same transaction as the state change.
- If the id already exists, acknowledge and return early.
- Never let a duplicate advance the state machine twice.

## Track state, not events

The dashboard doesn't care about individual webhooks — it cares about _where this fax
is right now_. I collapse the event stream into a single status per fax and keep an
audit log of transitions for support and compliance.

> The webhook is the messenger. The state machine is the truth.

## What made it reliable

- Signature verification on the very first line of the handler.
- Idempotency keyed on the provider's event id.
- A persisted state machine with a full audit trail.

None of it is glamorous, but together it's the difference between "usually works" and
"trust it with a patient record."
